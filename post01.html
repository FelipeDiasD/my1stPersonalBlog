GIT e versionamento de código

Versionamento de código e o Git em si acredito serem tópicos essenciais para quem está aos poucos entrando no mundo da programação. Neste post vou explicar o que é git e como funciona o versionamento de código.

Versionamento

O que seria esse tal versionamento? Bem, quando se fala em versionamento, já se pode fazer a ligação com a palavra versão. Não é incomum que você já tenha feito um trabalho de faculdade ou no ensino médio e teve que realizar várias alterações em arquivos em que, na teoria, eram as versões finais, resultando na seguinte situação:

O versionamento vem de encontro com essa situação ao se ter em um sistema, uma forma de se ter todos os registros de alterações realizadas em um determinado arquivo, nos dando a possibilidade de voltar em uma versão anterior, ou adicionar alguma alteração sem ter que necessariamente salvar algo separadamente. Adicionalmente, isso também nos dá a possibilidade de que possamos trabalhar em equipe de maneira mais coesa, ao partirmos de uma mesma versão de determinado arquivo e cada um adicionar uma alteração diferente e assim no final conseguindo unir essas versões em uma só com todas as alterações. No nosso caso, esse conceito é aplicado um arquivo de código.

O que é Git?

Git é um sistema de versionamento de código, que aplica o conceito explicado acima, de forma que o mesmo guarda os registros de versão do código como snapshots (“tira uma foto” de como o arquivo estava no momento em que foi salvo no sistema), salvando também a referência/caminho para o mesmo. Sendo assim, se forma um fluxo de snapshots.

Isso difere o git de outros sistemas de versionamento, pois outros sistemas se utilizam de um conceito de versionamento baseado em arquivos em si, e não em um snapshot. O que gera um fluxo de arquivos e modificações nesse arquivo.

Outro ponto de destaque do Git é o fato de que ele opera localmente. Todas as operações e gravações realizadas pelo git estão localmente no seu disco, fazendo com que a maioria seja quase instantânea. Isso também é um salva vidas quando se está offline, seja num avião ou mesmo se a internet caiu.

Funcionamento do Git

Mas onde o Git guarda esses dados? Bem, para isso o Git cria um repositório, que nada mais é que a base de dados onde estarão todas suas alterações. E para funcionar ele utiliza uma lógica que atribui estados para os arquivos. Tais estados são:

  • Modified: O arquivo em questão foi alterado, mas ainda não foi feito o commit ou seja, registro dessa alteração na base de dados.
  • Staged: As alterações realizadas no arquivo em questão foram marcadas para serem salvas/registradas no próximo commit ou seja, na próxima vez que o registro na base de dados for realizado.
  • Commited: Os dados da alteração realizada no arquivo estão registrados e salvos na base de dados.

Com esses estados em mente, no Git temos três seções principais:

  • Working tree: é basicamente onde realizamos as alterações. Os dados são puxados do repositório (Git repository) e disponibilizados para realizarmos qualquer alteração.
  • Staging area: Aqui estarão os dados marcados para serem registrados/salvos no próximo commit.
  • Git repository: Esta é a parte mais importante do sistema, o seu repositório, aquele que comentamos no qual estão salvos todos os dados e metadados sobre os seus arquivos e projeto como um todo.

Introdução ao uso do Git

Para usar o nosso querido Git é primeiramente necessário que você instale ele em seu computador. Sendo assim, no caso eu encorajo a ler o guia de instalação no site oficial, é bastante simples pricipalmente se você utiliza Windows. O maneira principal de você realizar seu verisionamento pelo Git é utilizando a CLI (comand line interface ou interface de linha de comando). Com o mesmo já instalado você irá até o diretório que você deseja versionar e irá clicar com botão direito do mouse e selecionar a seguinte opção para abrir a CLI:

A partir daí a uma interface de comando semelhante ao CMD deve se abrir para você. De cara é recomendado realizar alguns primeiros comandos para configurar seu usuário dentro do Git. Para isso insira os comandos:

  • git config --global user.name "Coloque seu username aqui" - Define o username do usuário no sistema Git
  • git config --global user.email "Insira um e-mail válido nessa configuração ()" - Define o e-mail que será utilizado pelo usuário em questão
  • git config --global init.defaultBranch main
Neste último comando você pode ter reparado que configuramos algo chamado branch, mais especificamente a default branch (ou branch padrão). As branches (ou ramificações) são usadas no contexto de versionamento para realizar alterações fora da linha principal de alterações. Por exemplo, há um fluxo de alterações numa página de um site e uma nova função será adiconada. Muito provavelmente será criada uma ramificação ou branch diferente para se trabalhar, na qual haverá apenas as alterações relacionadas a esta nova função. Depois do desenvolvimento e aprovação da mesma é possível fundir essas ramificações (o famoso branch merge).

No caso explicamos brevemente o que é uma branch, mas não o comando em si rs. No caso, o último comando nomeia qual o nome da sua branch principal, aquela que vai existir desde o começo. Os nomes mais comuns são master ou main (que é definido na última parte do comando), aí vai da escolha do cliente (eu prefiro master, por exemplo)

Para finalmente você começar a usar o Git e ir com tudo pra cima do versionamento é só utilizar o comando:

  • git init - Este comando vai inicializar um repositório no diretório que você abriu a CLI
A partir daí só resta você saber mais algumas coisinhas para iniciar o versionamento do seu código com sucesso. Todo arquivo adicionado ao diretório de trabalho do git tem dois estados (sim dois fora aqueles que já apresentamos), que determinam se o Git está "monitorando" os mesmos, tais são - Tracked (monitorad) ou Untracked (não monitorado). Para que um arquivo não monitorado seja adicionado ao "monitoramento" do Git, fazendo com que ele consiga ser versionado precisamos do seguinte comando:
  • git add "nome do arquivo do diretório"
    • Ou
  • git add .
    • Neste último comando, adicionamos todos os arquivos para serem monitorados pelo Git
Com isso complementamos o que abordamos sobre estados dos arquivos dentro do Git e para isso trago uma nova imagem que representa o fluxo mais detalhado do git.

No fim só faltou dizer como que nós realizamos o commit, para isso usamos esse comando intuitivo:

  • git commit -m "mensagem de commit"
  • git commit
    • Nesse útimo comando o Git irá abrir uma janela com um editor de texto para que possamos escrever a mensagem da nossa alteração (por padrão é o VIM)
Assim completamos uma introdução ao uso do git localmente. Um pouco por trás dos panos:

Efetivamente, esse snapshot que citamos é um hash code resultado de um checksum do seu arquivo momento em que ele foi salvo. Este hash é uma string de 40 caracteres composta de caracteres hexadecimais (0-9 e a-f) e calculada com base no conteúdo de um arquivo ou estrutura de diretório no Git.
